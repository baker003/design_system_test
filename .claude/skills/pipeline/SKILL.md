# Pipeline Orchestrator

사용자의 컴포넌트/기능 요청을 받아 에이전트 팀을 순서대로 조율합니다.
파이프라인은 과거 결정을 먼저 읽고(Memory Read), 작업을 수행한 뒤, 이번 결정을 남기며(Memory Write) 끝납니다.

## 참조 문서
- `.claude/doc/project-brief.md` -- 서비스 정의
- `.claude/doc/tech-spec.md` -- 기술 스택
- `.claude/doc/design-system.md` -- 디자인 시스템

## 메모리 저장소
- `.claude/memory/principles.md` -- 반복된 결정에서 승격된 도메인별 원칙
- `.claude/memory/decision-log.jsonl` -- 작업별 핵심 결정 누적 로그 (한 줄 = 결정 하나)

---

## 파이프라인 플로우

### [1/9] Memory Read
코드를 작성하기 전에, 과거의 결정을 먼저 읽어 컨텍스트에 주입합니다.
- `.claude/memory/principles.md`를 읽어 누적된 원칙 전체를 로드하세요. (파일이 없으면 건너뜀)
- `.claude/memory/decision-log.jsonl`에서 이번 요청과 관련된 항목을 찾으세요. 매칭 기준: 요청에 등장하는 도메인/컴포넌트 키워드가 로그의 `domain` 또는 `component` 또는 `topic`에 포함되는 항목. (파일이 없거나 매칭이 없으면 건너뜀)
- 로드한 원칙과 관련 결정을 **요약하여 이후 모든 단계(특히 PM·Designer)에 함께 전달**하세요.
- 완료 후 보고: `"[1/9] Memory Read 완료 -- 적용 원칙 {N}건, 참고 결정 {M}건"`

---

### [2/9] Project Manager

**범위 검증 3문** — PM 에이전트 호출 전, 오케스트레이터가 아래 3가지를 먼저 답합니다.
이 단계를 건너뛰면 명세가 과도하게 커지거나 이미 있는 컴포넌트와 중복됩니다.

| # | 질문 | 판단 기준 |
|---|------|---------|
| Q1 | 이 컴포넌트/기능이 어떤 구체적인 화면·플로우에서 쓰이나? | 시나리오가 없으면 → 사용자에게 확인 후 진행 |
| Q2 | 현재 DS_2 컴포넌트 조합으로 충족되지 않나? | 조합 가능하면 → 신규 컴포넌트 대신 사용법 안내 |
| Q3 | MVP에 반드시 필요한 variant는 무엇인가? | MVP 외 variant는 명세에서 제외 |

3문 통과 후 `project-manager` 에이전트를 호출하세요.
- 사용자의 요청 + Memory Read 원칙·결정 + 범위 검증 3문 답변을 함께 전달
- 완료 후 보고: `"[2/9] PM 완료 -- {명세 요약}"`

---

### [3/9] PM 승인 게이트
PM의 명세를 사용자에게 보여주고 승인을 요청하세요.
- **승인** → [4]로 진행
- **수정 요청** → PM 에이전트 재호출

---

### [4/9] Designer
`designer` 에이전트를 호출하세요.
- PM의 승인된 명세 + Memory Read 원칙을 전달
- **반드시 Figma를 직접 조회하여 치수·색상·상태를 확인하고 그대로 설계에 반영** (추정값 사용 금지)
- Designer가 컴포넌트 설계를 작성하여 반환합니다
- 완료 후 보고: `"[4/9] Designer 완료 -- {설계 요약}"`

---

### [5/9] Designer 승인 게이트
설계를 사용자에게 보여주고 승인을 요청하세요.
- **승인** → [6]으로 진행
- **수정 요청** → Designer 에이전트 재호출

---

### [6/9] Frontend
`frontend-dev` 에이전트를 호출하세요.
- Designer의 승인된 설계를 전달 (Figma 스펙을 그대로 적용, 임의 수정 금지)
- **아이콘 작업이 포함된 경우**: `icon-designer` 에이전트를 별도로 호출 (general-purpose로 직접 그리지 않음)
- 완료 후 보고: `"[6/9] Frontend 완료 -- {구현 요약}"`

---

### [7/9] Compliance Check
변경된 파일을 대상으로 아래 규칙을 grep/find로 자동 채점합니다.

| 코드 | 규칙 | 검사 명령 |
|------|------|---------|
| C1 | 색상 하드코딩 금지 | `grep -rn "#[0-9a-fA-F]\{3,6\}"` → 시맨틱 토큰 미사용 여부 |
| C2 | tailwind.config.ts 생성 금지 | `find . -name "tailwind.config.ts"` |
| C3 | 'use client' 최소화 | 변경 파일 내 'use client' 선언 개수 확인, 불필요 여부 판단 |
| C4 | Props 인터페이스 명시 | 컴포넌트 파일에 `interface.*Props` 또는 `type.*Props` 정의 여부 |
| C5 | 파일당 ~300줄 권장 | `wc -l` 로 확인, 300줄 초과 시 경고 |
| C6 | 시맨틱 토큰 사용 | `--color-`, `--spacing-` 등 CSS 변수 사용 여부 |

- **COMPLIANCE: FAIL** → frontend-dev에게 위반 목록 전달, 수정 후 재채점 (최대 2라운드)
- **COMPLIANCE: PASS / PASS(경고)** → [7.5]로 진행
- 완료 후 보고: `"[7/9] Compliance 완료 -- {PASS/FAIL}, 위반 {V}건"`

---

### [7.5/9] Figma 시각 대조 (Figma 노드 기반 구현 시 필수)
오케스트레이터가 직접 수행합니다. Reviewer 호출 전 반드시 완료합니다.

1. `figma-sync` 에이전트로 구현 대상 노드의 **스크린샷을 캡처**합니다
2. 구현된 코드에서 모든 좌표·크기·색상 값을 추출합니다
3. 아래 표를 채워 불일치를 확인합니다:

| 요소 | Figma 값 | 코드 값 | 일치 여부 |
|------|----------|---------|----------|
| (각 요소 나열) | | | ✅/❌ |

4. ❌ 항목이 있으면 frontend-dev에게 수정 요청 후 재대조합니다
5. 모든 항목 ✅일 때만 [8]로 진행합니다
- 완료 후 보고: `"[7.5/9] Figma 대조 완료 -- 불일치 {N}건"`

---

### [8/9] Reviewer
`reviewer` 에이전트를 호출하세요.
- **APPROVED** → [9]로 진행
- **CHANGES_REQUESTED** → frontend-dev에게 수정 요청 후 재리뷰 (최대 3라운드)
- 완료 후 보고: `"[8/9] Reviewer 완료 -- {리뷰 결과}"`

---

### [9/9] Memory Write
이번 작업에서 내려진 핵심 결정을 메모리에 남깁니다.
- 이번 파이프라인에서 실제로 내려진 **설계·구현 결정**을 추출하세요.
  추출 대상: 컴포넌트 선택, 색상/토큰 선택, 레이아웃 방식, 상태 처리 방식, 승인 게이트에서 사용자가 교정한 내용.
- 각 결정을 아래 형식의 JSON 한 줄로 `.claude/memory/decision-log.jsonl`에 **append** (기존 내용 덮어쓰기 금지):
  `{"date":"YYYY-MM-DD","domain":"{도메인}","component":"{컴포넌트명}","topic":"{결정 주제}","decision":"{무엇을 결정했는지}","rationale":"{왜}"}`
- **원칙 승격 검사**: 같은 `topic`에 대한 동일한 `decision`이 3회 이상 반복되면, `principles.md` 승격 후보로 사용자에게 제안하세요. (사용자 승인 없이 principles.md 수정 금지)
- 완료 후 보고: `"[9/9] Memory Write 완료 -- 기록 {K}건, 승격 제안 {P}건"`

## Figma 반영 (선택, 사용자 요청 시만)

Figma 반영은 사용자가 명시적으로 요청한 경우에만 진행합니다. 요청이 없으면 건너뜁니다.

### Figma 사전 조사 (figma-sync 호출 전 오케스트레이터 직접 실행)

**① 기존 컴포넌트 확인**
`mcp__figma__get_figma_data`로 파일 내 페이지 목록을 읽어 동일 이름 페이지가 있는지 확인한다.
- 이미 있으면 → 해당 node ID 추출 → figma-sync에 "기존 노드 업데이트" 모드로 전달
- 없으면 → 신규 생성 모드

**② 레퍼런스 노드 구조 읽기 (신규 생성 시)**
기존 컴포넌트 섹션 1개를 읽어 문서 프레임 구조(크기, 여백, 텍스트 스타일)를 추출하고 figma-sync 프롬프트에 포함한다.

**③ 사용할 아이콘 키 확인**
컴포넌트에 필요한 아이콘 키를 사전에 파악해 프롬프트에 명시한다.

**④ 관련 DS2 컴포넌트 키 확인**
인스턴스로 배치할 DS2 컴포넌트(Button, Chip 등)가 있으면 해당 ComponentSet 키를 파악해 프롬프트에 포함한다.

### Figma 반영 — 컴포넌트 1개 = 에이전트 1개

각 figma-sync 에이전트 호출 시 프롬프트에 반드시 포함:
- 대상 node ID (부모 섹션 또는 페이지)
- 배치 좌표 (x, y)
- 레퍼런스 노드 구조 수치
- 사용할 아이콘 키
- 사용할 DS2 컴포넌트 키 (있는 경우)
