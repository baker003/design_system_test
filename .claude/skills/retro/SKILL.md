# Retro — 스프린트 회고

작업 종료 후 git 기록과 대화 컨텍스트를 기반으로 회고를 작성합니다.
무엇을 했는지 나열하는 게 아니라, 패턴을 찾고 다음 스프린트를 개선하는 것이 목적입니다.
(gstack `/retro` 패턴을 DS_2 맥락에 맞게 재해석)

---

## 실행 절차

### Step 1 — 작업 수집
아래 명령으로 해당 기간의 커밋을 수집합니다:

```bash
# 최근 N개 커밋
git log --oneline -20

# 날짜 범위 지정
git log --oneline --after="2026-06-01" --before="2026-06-08"

# 파일별 변경 요약
git diff --stat HEAD~10
```

### Step 2 — 카테고리 분류
수집한 커밋과 작업을 아래 카테고리로 분류합니다:

| 카테고리 | 설명 |
|---------|------|
| ✅ 완료 | 계획대로 완료된 작업 |
| 🔁 반복 수정 | 같은 부분을 2회 이상 수정한 작업 |
| 🚧 미완료 | 시작했지만 완료하지 못한 작업 |
| ⚠️ 예상 외 이슈 | 계획에 없었던 문제 대응 |

### Step 3 — 패턴 분석
🔁와 ⚠️ 항목을 중심으로 패턴을 찾습니다:
- 같은 유형의 실수가 반복됐나?
- 특정 단계(Designer/Frontend/Reviewer 등)에서 막히는 패턴이 있나?
- 추정이 틀렸던 부분은 어디인가?

### Step 4 — 다음 스프린트 액션
패턴에서 도출된 구체적인 변경 사항을 정리합니다. (최대 3개, 실행 가능한 것만)

---

## 출력 형식

```markdown
## 스프린트 회고 — {기간}

### 완료
- ...
- ...

### 반복 수정된 작업
- {작업명}: {몇 번 수정했는지, 이유}

### 미완료
- {작업명}: {막힌 이유}

### 예상 외 이슈
- {이슈}: {어떻게 해결했는지}

---

### 패턴
1. (반복되는 실수나 막히는 지점)
2. ...

### 다음 스프린트 액션 (최대 3개)
- [ ] ...
- [ ] ...
- [ ] ...
```

---

## 언제 사용하나

- 스프린트 마무리 시
- 주간 회고 미팅 전
- 같은 실수가 반복된다고 느낄 때
- 작업 방식을 개선하고 싶을 때
