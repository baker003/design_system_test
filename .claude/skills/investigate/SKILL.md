# Investigate — 근본 원인 조사

버그나 이상 현상을 발견했을 때 바로 수정하지 않습니다.
먼저 무엇이 왜 일어났는지 파악하고, 수정 범위를 확정한 뒤 작업합니다.
(gstack `/investigate` 패턴을 DS_2 Next.js 맥락에 맞게 재해석)

---

## 원칙

> **조사 없이 수정 금지.** 증상만 보고 코드를 바꾸면 근본 원인이 남아 같은 버그가 다시 나타납니다.

---

## 조사 절차

### Step 1 — 증상 정확히 기술
- 어떤 화면에서, 어떤 동작을 했을 때, 어떤 결과가 나왔나?
- 재현 조건은? (항상/특정 조건만/간헐적)
- 에러 메시지나 콘솔 로그가 있으면 전문 기록

### Step 2 — 영향 범위 파악
- 어떤 컴포넌트/파일이 관련되는가?
- `grep`으로 관련 심볼·클래스·함수명을 검색하여 참조 지점 목록화
- 최근 변경 이력 확인: `git log --oneline -20 -- {파일명}`

### Step 3 — 가설 수립 (최소 2개)
- 가능한 원인을 2가지 이상 나열합니다
- 각 가설이 증상을 설명할 수 있는지 검토합니다

### Step 4 — 가설 검증
- 각 가설을 코드 읽기(Read)나 grep으로 확인합니다
- 로그·테스트·빌드 결과로 가설을 좁힙니다
- 근본 원인이 특정되면 Step 5로 진행합니다

### Step 5 — 수정 범위 확정
- 근본 원인: (한 문장으로 명시)
- 수정 대상 파일: (목록)
- 예상 사이드 이펙트: (있다면 명시, 없으면 "없음")
- 수정 후 검증 방법: (어떻게 고쳐졌는지 확인할 것인지)

### Step 6 — 수정 및 검증
Step 5를 사용자에게 공유하고 승인받은 뒤 수정합니다.
수정 후 Step 5에서 정한 검증 방법으로 확인합니다.

---

## 출력 형식

```
## 조사 보고서

**증상**: ...
**재현 조건**: ...

### 가설 목록
| # | 가설 | 검증 결과 |
|---|------|---------|
| 1 | ... | 기각 / 확인 |
| 2 | ... | 기각 / 확인 |

### 근본 원인
(한 문장)

### 수정 계획
- 파일: ...
- 변경 내용: ...
- 사이드 이펙트: ...
- 검증 방법: ...
```

---

## 언제 사용하나

- 버그 리포트가 들어왔을 때
- "왜 이렇게 동작하지?" 원인 불명 이슈
- 수정했는데 같은 버그가 재발할 때
- 수정 전 영향 범위가 불확실할 때
