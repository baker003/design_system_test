# Office Hours — 제품 전략 검토

새 기능을 만들거나 방향을 바꾸기 전에 실행합니다.
6가지 질문에 솔직하게 답하면서 아이디어를 검증하고, 필요하지 않은 작업을 걸러냅니다.
(gstack `/office-hours` 패턴을 DS_2 맥락에 맞게 재해석)

---

## 실행 방법

아래 6가지 질문을 순서대로 사용자에게 물어봅니다.
각 답변을 받은 뒤 다음 질문으로 넘어갑니다. 답변 없이 건너뛰지 않습니다.

---

## 6가지 강제 질문

### Q1. 실제 수요: 누가, 언제 이것을 쓰나?
구체적인 사용자와 상황을 말해주세요.
> "사용자가 쓸 것 같다"는 가정이 아니라, 지금 실제로 겪고 있는 불편이 있나요?

### Q2. 현황: 지금은 어떻게 해결하고 있나?
현재 이 문제를 다른 방법으로 우회하고 있나요?
> 우회 방법이 있다면, 새 기능이 그것보다 얼마나 더 나은가요?

### Q3. 절실함: 이게 없으면 어떻게 되나?
이 기능 없이 제품이 동작하지 않는 경우인가요?
> 없어도 괜찮으면 → 지금 당장 만들 필요가 없습니다.

### Q4. 최소 범위: 가장 작은 버전은 무엇인가?
전체 스펙이 아닌, 지금 당장 검증할 수 있는 MVP 범위를 한 문장으로 말해주세요.
> 이 MVP 밖의 모든 것은 다음 이터레이션으로 미룹니다.

### Q5. 기존 자산 활용: 지금 있는 것으로 안 되나?
DS_2 기존 컴포넌트, 기능, 페이지 조합으로 해결되지 않나요?
> 새로 만드는 것보다 기존 것을 조합하는 게 더 빠를 수 있습니다.

### Q6. 미래 부담: 이 결정이 나중에 발목을 잡을 수 있나?
이 기능을 추가하면 나중에 어떤 기술 부채나 설계 변경이 생길 수 있나요?
> 부담이 크다면 → 지금이 올바른 타이밍인지 재검토합니다.

## 출력 형식

6문 답변을 모두 받은 뒤 아래 형식으로 요약합니다:

```
## Office Hours 결과

| 질문 | 답변 요약 | 판정 |
|------|---------|------|
| Q1 실제 수요 | ... | ✅ / ⚠️ / ❌ |
| Q2 현황 | ... | ✅ / ⚠️ / ❌ |
| Q3 절실함 | ... | ✅ / ⚠️ / ❌ |
| Q4 최소 범위 | ... | ✅ / ⚠️ / ❌ |
| Q5 기존 자산 | ... | ✅ / ⚠️ / ❌ |
| Q6 미래 부담 | ... | ✅ / ⚠️ / ❌ |

### 종합 판정
- ✅ 5개 이상: 진행 권장
- ⚠️ 3~4개: 범위 축소 후 진행 권장
- ❌ 3개 이상: 재검토 또는 보류 권장

### 권장 MVP 범위
(Q4 답변 기반으로 최소 구현 범위 명시)

### 다음 단계
(pipeline 스킬 실행 / 추가 조사 / 보류 중 제안)
```

## 언제 사용하나

- 새 컴포넌트나 기능 추가 결정 전
- 기존 기능의 대규모 변경 전
- "이거 만들어야 할까?" 불확실할 때
- `/pipeline` 실행 전 선택적으로 실행
